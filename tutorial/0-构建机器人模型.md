# 0-构建机器人模型

官方参考文档：https://nvlabs.github.io/curobo/latest/getting-started/build_robot_model.html

本节目标是从自定义机械臂 URDF 出发，生成 cuRobo 可用的 robot model 配置文件：

- `mec_arm_from_urdf.yml`：cuRobo 原生 robot config
- `mec_arm_from_urdf.xrdf`：Isaac Sim / Isaac Lab 生态更常用的 robot description 格式

cuRobo 不能只依赖普通 URDF 做运动规划。URDF 主要描述 link、joint、mesh、惯量和关节限制，但 cuRobo 还需要两类额外信息：

- 每个 link 的 collision spheres，用于快速碰撞检测
- self-collision ignore matrix，用于跳过相邻 link 或不需要检查的自碰撞对

cuRobo 官方教程中的 `RobotBuilder` 会从 URDF 和 mesh 自动拟合 collision spheres，并生成 self-collision ignore matrix。

## Step 4: Understand the pipeline

官方 builder 主要做三件事。

第一，sphere fitting。

cuRobo 会把每个 link 的 mesh 近似成若干 collision spheres。后续碰撞检测主要基于这些球体，而不是直接对复杂三角 mesh 做实时距离查询。这样可以大幅提高规划速度。

第二，self-collision matrix。

机械臂中很多 link 对不需要每次都检查碰撞。例如相邻 link 通常天然接触或距离很近，如果不忽略，会导致规划器认为机械臂一直处于自碰撞状态。builder 会通过随机采样关节状态，生成一份 self-collision ignore matrix。

第三，export。

builder 会把 URDF 解析出的运动学树、joint limits、tool frame、collision spheres 和 self-collision ignore matrix 写入配置文件。YAML 是 cuRobo 原生格式，XRDF 更适合和 Isaac Sim / Isaac Lab 工作流对接。

## Step 5: Tuning

第一版先保留默认参数：

```bash

--num-collision-samples 1000
```

后续如果 collision spheres 可视化效果不好，再调这些参数：

```text
--sphere-density
  提高或降低每个 link 的碰撞球数量。数值越大，覆盖更细，但规划计算也更重。

--coverage-weight
  提高 mesh 内部覆盖程度。覆盖不足时可以调大。

--protrusion-weight
  降低球体伸出 mesh 外部的程度。球体外扩太严重时可以调大。

--clip-link base_link z 0.0
  如果 base_link 的球体穿过安装平面或桌面，可以裁剪 base_link 的 z 方向球体。

--num-collision-samples
  增加 self-collision matrix 采样数量。复杂机械臂或自碰撞结果不稳定时可以调大。
```

### 为什么 default_joint_state 不是全零

脚本输出中的默认关节角是：

```text
[0.0, -1.2999999523162842, 2.0, 0.0, 1.5499999523162842, 0.0]
```

它来自 `mec_arm_from_urdf.yml` 里的：

```yaml
cspace:
  default_joint_position:
  - 0.0
  - -1.2999999523162842
  - 2.0
  - 0.0
  - 1.5499999523162842
  - 0.0
```

这个值不是 Isaac Sim 当前关节角，也不是 URDF 零位，而是 cuRobo 的默认参考姿态。它会用于 IK seed、retract posture、null-space bias 或 trajectory optimization regularization。

这个默认姿态刚好接近各关节上下限的中点：

```text
Joint1: [-3.1, 3.1]   -> 0.0
Joint2: [-2.6, 0.0]   -> -1.3
Joint3: [0.0, 4.0]    -> 2.0
Joint4: [-3.1, 3.1]   -> 0.0
Joint5: [0.0, 3.1]    -> 1.55
Joint6: [-1.57, 1.57] -> 0.0
```

相比之下，全零姿态对这台机械臂不是一个好的默认姿态，因为：

```text
Joint2 = 0.0  位于上限
Joint3 = 0.0  位于下限
Joint5 = 0.0  位于下限
```

规划器通常不希望默认姿态贴着关节限位。后续如果要改 `default_joint_position`，应该选择一个远离关节上下限、无明显自碰撞、TCP 姿态合理的 home pose，而不是简单改成全零。

### sys.path.insert

```python

    sys.path.insert(0, str(CUROBO_SOURCE_ROOT))
```

这行代码让 Python 优先从 `/home/light/workspace/curobo` 导入 cuRobo 源码。你的环境里 `nvidia-curobo` 包版本号显示为 `0.0.0`，而实际可用的新 API 来自本地源码，所以这里显式把源码路径放到 `sys.path` 最前面。

### cuRobo API: MotionPlannerCfg

```python
from curobo.motion_planner import MotionPlannerCfg
```

`MotionPlannerCfg` 是 cuRobo 的规划器配置入口。它会把这些信息整合成一个配置对象：

- robot yml
- IK solver 配置
- trajectory optimization 配置
- graph planner 配置
- scene/world 配置
- self-collision 配置
- CUDA/device 配置

本脚本里使用：

```python
cfg = MotionPlannerCfg.create(
    robot=str(ROBOT_CONFIG),
    scene_model=None,
    self_collision_check=True,
    use_cuda_graph=False,
    num_ik_seeds=4,
    num_trajopt_seeds=1,
)
```

参数含义：

- `robot`：加载 `mec_arm_from_urdf.yml`。
- `scene_model=None`：不加载外部障碍物，只验证机器人模型。
- `self_collision_check=True`：启用自碰撞配置，验证 `self_collision_ignore` 能正常参与配置构建。
- `use_cuda_graph=False`：调试阶段关闭 CUDA graph，减少缓存和重复运行带来的限制。
- `num_ik_seeds=4`：IK 求解器使用的 seed 数量。当前脚本不做 IK，但 MotionPlannerCfg 仍会创建相关配置。
- `num_trajopt_seeds=1`：轨迹优化 seed 数量。当前脚本不做 trajopt，只保留最小配置。

### cuRobo API: MotionPlanner

```python
from curobo.motion_planner import MotionPlanner
```

`MotionPlanner` 是 cuRobo 的主入口对象。后续 FK、IK、轨迹规划都会通过它进入。本脚本里用它做三件事：

- 读取 `planner.joint_names`
- 读取 `planner.tool_frames`
- 调用 `planner.compute_kinematics(joint_state)` 做 FK

脚本结束时调用：

```python
planner.destroy()
```

这是为了显式释放 MotionPlanner 内部可能持有的 CUDA、碰撞检测、IK、trajopt 资源。频繁调试时建议保留这个清理动作。

### cuRobo API: JointState

```python
from curobo.types import JointState
```

`JointState` 是 cuRobo 表示关节状态的数据结构。它不是单纯的 Python list，而是包含：

- `position`
- `velocity`
- `acceleration`
- `jerk`
- `joint_names`
- `device_cfg`

本脚本使用：

```python
joint_state = JointState.from_position(
    position=q_default,
    joint_names=list(planner.joint_names),
)
```

这里要注意 `position` 的形状。cuRobo FK 期望输入是 batch 形式：

```text
[batch, dof]
```

所以如果读到的默认关节角是：

```text
[dof]
```

脚本会用：

```python
q_default = q_default.unsqueeze(0)
```

把它变成：

```text
[1, dof]
```

### run_fk

```python
def run_fk(planner: MotionPlanner, joint_state: JointState):
    kin_state = planner.compute_kinematics(joint_state)
    ...
```

这个函数执行 cuRobo FK。

核心 API 是：

```python
kin_state = planner.compute_kinematics(joint_state)
```

返回的 `kin_state` 是一个 `KinematicsState`，主要包含：

- `tool_poses`：tool frame 的位姿
- `tool_jacobians`：tool frame 的雅可比
- `robot_spheres`：当前关节姿态下的 collision spheres
- `robot_collision_geometry`：机器人碰撞几何描述

本脚本用：

```python
tcp_pose = kin_state.tool_poses.get_link_pose(
    EXPECTED_TOOL_FRAME,
    make_contiguous=True,
)
```

从 `tool_poses` 中取出 `TCP_link` 的位姿。

`tcp_pose.position` 是：

```text
TCP_link 在 base_link frame 下的位置，单位米
```

`tcp_pose.quaternion` 是：

```text
TCP_link 在 base_link frame 下的姿态四元数，顺序为 wxyz
```

这和 Isaac Sim 里常见的 USD world pose 不同。以后做 Isaac-vs-cuRobo 对齐时，需要把 Isaac 的 TCP world pose 转换到 base frame：

```text
T_base_tcp = inverse(T_world_base) @ T_world_tcp
```

`kin_state.robot_spheres` 的形状是：

```text
[batch, horizon, num_spheres, 4]
```

最后一维是：

```text
[x, y, z, radius]
```

这些 spheres 位于 robot base frame 下，是后续 cuRobo 碰撞检测的机器人本体近似。

### main

```python
def main():
    planner = None
    try:
        planner = create_planner()
        check_robot_metadata(planner)
        joint_state = get_default_joint_state(planner)
        run_fk(planner, joint_state)
    finally:
        if planner is not None:
            planner.destroy()
```

`main()` 是脚本入口，按顺序执行完整检查链路：

```text
加载配置
-> 创建 planner
-> 检查 joint_names/tool_frames
-> 构造 default JointState
-> 执行 FK
-> 释放 planner
```

如果 `main()` 最后打印：

```text
mec_arm_from_urdf.yml 可以被 cuRobo 加载，并且可以计算 TCP_link FK。
```

说明当前 robot model 已经通过第一轮 cuRobo 侧验证。
